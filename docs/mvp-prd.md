# 亲密空间小程序 MVP 方案

> 这份文档只保留产品目标和规划。当前工程实现、页面、云函数、数据字段与微信 API 对照见 `docs/project-reference.md`。

## 1. 目标与边界

- 平台优先级：先上线微信小程序。
- 关系模型：一个用户只能拥有或加入 1 个空间。
- 解绑策略：解绑后历史数据不保留。
- 当前实现范围：空间创建、邀请同意、任务创建、任务完成、回忆归档、解绑。
- 当前已删除/暂缓：类/Category、独立 `category-service`、独立 `theme-service`。

## 2. 核心对象

### Space

双人关系容器。创建后处于 `pending`，对方同意后变为 `active`。

### Task

空间内的执行事项。MVP 当前支持标题、地点文本、图片地址、截止日期、完成状态。

### Memory

当前不是独立集合，而是由 `completed` / `overdue` 任务派生出的回忆时间线。

### Score / Theme

亲密度是隐藏分，不直接展示数值；当前由 `space-service` 按分数映射空间主题。

## 3. 关键流程

1. 首次进入：无空间时引导创建空间。
2. 创建空间：生成 pending 空间和邀请码。
3. 邀请同意：当前为模拟同意，后续应改为 token 邀请。
4. 创建任务：active 空间下创建 todo 任务。
5. 完成任务：任务变为 completed，并进入回忆时间线。
6. 解绑空间：删除空间，后续需补任务级联清理。

## 4. 已确认规则

1. 每个用户只能有一个空间。
2. 解绑后历史不保留。
3. 图片凭证可选。
4. 定位默认尝试获取，也可不填。
5. 任务完成不需要另一方确认，但另一方可查看。
6. 逾期扣分后续自动结算。
7. 亲密度隐藏，通过背景/主题表现。
8. 后续需要提醒能力，包括小程序订阅消息。

## 5. 后续规划

### P1：业务正确性

- 邀请按 `inviteToken` 查询和确认。
- 限制一个用户只能有一个空间。
- 完成任务、解绑空间时补权限和归属校验。
- 解绑时级联清理任务。

### P2：体验补齐

- 图片凭证改为选择媒体 + 云存储。
- 地点改为定位/地图选择。
- 订阅消息提醒任务和邀请。
- 回忆归档支持筛选。

### P3：规则系统

如果后续仍需要“类/Category”，建议重新设计后再恢复：

- 类创建和双方同意。
- 结构化规则配置。
- 任务按规则加减分。
- 分数流水 `score_ledger`。

## 6. 建议里程碑

1. M1：空间创建、邀请、同意跑通。
2. M2：任务创建、完成、回忆归档跑通。
3. M3：权限校验、单空间限制、解绑级联清理。
4. M4：图片、地点、订阅消息体验补齐。
5. M5：按需要恢复类/规则/积分流水。
